# 后续剧情设计原则与模板

## 1. 目标体验

后续剧情应把当前“内容很多但彼此散落”的江湖，逐步组织成玩家能够理解和记住的个人历程：

> 我从哪里来，拜入何门，因何卷入此事，选择帮助谁，付出了什么，江湖后来如何谈论我。

新内容优先增加因果、选择和回响，不以对白长度、战斗数量或奖励稀有度代替剧情质量。

## 2. 核心原则

### 2.1 玩家必须是行动者

每个章节至少包含一个由玩家主动触发的目标和一个能改变结果的决定。原著人物可以提出问题、
制造压力或评价结果，但不能包办解决冲突。

### 2.2 尊重共享世界的可重复性

剧情结果优先写入玩家个人状态或实例场景。需要表现掌门死亡、城池陷落、门派决裂时，使用传闻、
梦境、回忆、阶段副本或可恢复的局部状态，不能让常驻功能永久失效。

### 2.3 先写角色动机，再挂经典名场面

引用原著桥段前必须回答：玩家为什么参与、各方现在想要什么、玩家有什么选择、完成后谁会记得。
只有复述原著台词而不改变玩家处境的内容，归为彩蛋或环境叙事，不列为主任务。

### 2.4 单人必可完成

真人组队、AI 同行、特定在线人数都只能提供支线或难度变化，不能成为主进度硬门槛。AI 缺席、
死亡、卡路或服务重启时，剧情必须能恢复或降级继续。

### 2.5 失败也要推进认知

失败可损失时间、资源、名望或某个分支机会，但应告诉玩家原因、保留已获得的信息，并提供重试、
替代路线或明确的永久后果。避免因隐藏资格或随机数让玩家无从理解地错过唯一内容。

### 2.6 奖励服务于身份

奖励优先顺序：关系与称号、剧情权限、独特但非破坏平衡的能力、纪念物、常规数值。高倍率单机版
已有充足成长资源，新剧情不应继续依靠巨额经验制造重要感。

### 2.7 控制语调

主线使用通俗武侠正剧；神话内容需要被传闻、梦境或稀有异象框定；现代梗和戏仿必须可选且不
阻断进度。一个章节内部不要无提示地跨越三种语调。

### 2.8 通用主线不接管师承

当前世界允许自由拜师，历史角色也大多已有师门。通用剧情可以读取当前师门生成称呼、线索和
回响，但不得调用 `set("family")`、`delete("family")`、`assign_apprentice()`，不得自动执行
拜师或叛师，也不得把无师门状态作为主线前置。玩家在剧情中途更换师承时，主进度继续有效，
后续文本按当前状态重新适配。

### 2.9 分支必须收束，禁止无界生长

章节选择默认产生“局部差异”，不得自动扩张为永久独立主线。不同选择可以改变进入下一阶段的
理由、同行人物、可用线索、解决手段、关系和回响，但应在预先指定的收束点重新进入共同骨干。
合流后必须继续承认旧选择，不能把状态清零或只改一句无关紧要的对白。

以下是强制预算：

- 单章 70% 至 80% 的流程应属于共同骨干，分支内容原则上不超过 20% 至 30%；
- 每章最多产生一个需要跨章读取的核心选择，且非必要不产生；其余选择应在本章内结算或折叠为关系变化；
- 每 3 至 5 章至少设置一个明确收束点，清理不再产生价值的临时分支状态；
- 普通章节不得新增永久独立主线；只有篇章级高潮经过专项评审后才允许例外；
- 一个大篇章最多保留两条真正独立的终局路线，且必须在篇章设计阶段明确必要性、预算和测试成本；
- 任何新分支都必须标注其收束章节、收束阶段和合流后仍保留的差异；没有收束方案的分支不进入实现。

允许长期保留的是少量有叙事价值的状态维度，例如民间关系、官面关系、商路关系、关键人物命运
和篇章级立场。不得永久保存每一次普通对话选项，也不得为所有历史选择制作笛卡尔积式组合文本。

### 2.10 收束不等于选择失效

不同路线到达同一事件时，至少通过下列一项继续体现旧选择：

- 由不同人物发出邀请或担保；
- 玩家拥有不同证据、资源、通行方式或非战斗解法；
- 相关人物的信任、敌意、称呼或出场顺序不同；
- 某条短支线、纪念物、传闻或结尾回响不同；
- 同一高潮中的代价、风险或救援条件不同。

如果一个选择在合流后既不改变状态、手段、关系，也没有可见回响，它就不应被包装成重大分支。

### 2.11 反派必须有明确恶意、计划与主动行动

主要反派不能只代表一种值得讨论的制度观点，也不能只等玩家调查。设计稿必须明确写出：

- 反派最终想获得什么权力、资源或结果；
- 反派已经主动伤害了谁，使用了什么不可接受的手段；
- 若玩家不介入，计划将在何时造成什么具体后果；
- 反派如何针对玩家的调查作出反制；
- 玩家在哪一章能够破坏计划并得到明确胜利反馈。

灰度应来自其他人物、组织历史、手段代价和受益者冲突，不能把主要反派的恶行写得含糊。反派
可以有可理解的理由，但理由不能抹去其蓄意制造伤害、剥夺选择或把人当工具的事实。

### 2.12 身份反转必须公平铺垫

每次重要身份反转至少提前提供两条相互独立的异常线索，并在揭晓后解释所有关键矛盾。禁止仅靠
“其实他一直是另一个人”制造惊讶。设计稿必须列出：

- 表面身份；
- 真实身份或真实立场；
- 反转前可见的两条以上线索；
- 用来误导玩家但最终能够合理解释的红鲱鱼；
- 揭晓后角色此前行为为何成立；
- 反转对玩法、关系或目标产生的实际影响。

同一章原则上只揭晓一个主要身份反转，避免连续翻转削弱可信度。

### 2.13 解谜必须可推理、可回看、可恢复

正式剧情每章至少包含一个非战斗推理或信息谜题。谜题必须遵守：

- 所有必需信息在游戏内出现，不要求玩家了解现实地理、历史或原著细节；
- 至少有两种信息来源，关键线索写入剧情日志并可随时回看；
- 规则在作答前展示，错误答案说明哪条事实冲突，不用无反馈试错；
- 不把随机数、隐藏属性或精确输入格式伪装成推理；
- 允许请求分级提示，提示只减少纪念性评价，不扣永久奖励；
- 断线、死亡、实例重建和关键物品丢失不会让谜题失去唯一解；
- 暴力可以改变场景压力，但不能跳过理解主线真相所需的核心推理。

复杂谜题最多同时追踪三至五项事实，适配移动端短时阅读。谜题答案确认后持久化，不能因重启
要求玩家重复作答。

## 3. 内容接入选择

| 需求 | 推荐载体 | 不推荐 |
| --- | --- | --- |
| 玩家随时推进的连续章节 | 专用任务守护 + 玩家持久状态 + 固定/实例场景 | 定时事件 |
| 强调时间、季节、地方风物 | `eventd` 地点事件 | 隐藏随机资格 |
| 稀有传闻、世界惊喜 | `storyd`，但需单机资格与错过保护 | 必经主线步骤 |
| 重复成长劳动 | 师门/公共任务框架 | 塞入唯一关键对白 |
| 多阶段战斗、场景会变化 | 实例副本 | 永久修改常驻主城 |
| 绝学与神兵奇遇 | 传闻入口 + 实体线索 + 固定 NPC/地点 + 永久标记 | 广播结束后直接无条件发奖 |
| 氛围和历史背景 | 房间、物件、NPC 问询、可选阅读 | 强制长篇频道刷屏 |

## 4. 建议的统一状态模型

现有状态键很分散。新增剧情建议统一放在玩家持久 dbase 的 `plot/` 命名空间：

```text
plot/<plot_id>/version       剧情数据版本
plot/<plot_id>/chapter       当前章节 ID
plot/<plot_id>/stage         当前阶段 ID
plot/<plot_id>/status        active/completed/failed/locked
plot/<plot_id>/choices       已确认选择 mapping
plot/<plot_id>/relations     本剧情相关人物关系变化 mapping
plot/<plot_id>/flags         少量持久事实 mapping
plot/<plot_id>/started_at    首次开始时间
plot/<plot_id>/updated_at    最近推进时间
plot/<plot_id>/completed_at  完成时间
```

约定：

- `temp/plot/...` 只存当前战斗、对话页或场景会话，不作为永久进度依据；
- 阶段推进应幂等，重复触发不能重复发唯一奖励；
- 先写新阶段与选择，再发奖励，异常后可判断是否已经结算；
- 物品不是唯一进度真相，丢失物品时可根据持久状态补发或提供替代；
- 旧 `story/*`、`guan/*`、`marks/*` 等状态需要复用时，在设计稿中逐项列出，不静默迁移；
- 每条剧情记录 `version`，为后续修复和存档迁移留入口。

## 5. 章节结构建议

一个标准中篇章节建议包含：

1. **钩子**：传闻、来信、故乡关系、师门命令或现场异象，说明为什么是玩家。
2. **调查**：至少从两个来源取得信息，允许玩家形成判断。
3. **压力**：时间、门规、资源、人物安全或立场冲突迫使玩家取舍。
4. **选择**：至少两个可理解选项，结果不能只差一行对白。
5. **行动高潮**：战斗、潜入、护送、说服、解谜或牺牲资源，可有非战斗解法。
6. **结算**：明确说明直接结果、奖励和损失。
7. **回响**：至少一个人物后续对白、传闻、称号、关系或新入口承认选择。
8. **收束声明**：说明本章各分支在何处合流，以及合流后保留哪些差异。

不是每条小任务都需要完整结构，但正式章节提案不得缺少“选择、结算、回响、收束声明”。

## 7. 提案审查清单

- 是否与“群侠并置、玩家自建侠客”的基线兼容？
- 是否明确区分原著事实、当前 zjmud 事实和新增设定？
- 玩家是否主动推进并至少做出一个有后果的选择？
- 是否保证一名真人可以完成？
- 是否会永久破坏师父、商店、交通或主城等共享功能？
- 是否说明死亡、掉线、重启和重复触发的恢复行为？
- 是否有可见日志，而不是只靠隐藏状态猜进度？
- 奖励是否避开当前单机高倍率造成的进一步膨胀？
- 是否至少有一次后续回响承认玩家选择？
- 是否满足共同骨干 70% 至 80%、分支 20% 至 30% 的预算？
- 是否每章最多只有一个需要跨章读取的核心选择，并说明了为何不能在本章内结算？
- 是否为每个跨章分支写明了收束章节、阶段和合流后保留的差异？
- 合流是否真正保留了关系、资源、手段或回响，而不是把选择悄悄清零？
- 是否避免把普通对话选项永久化并参与后续组合爆炸？
- 若产生永久独立路线，是否确属篇章级高潮、通过专项评审且未超过两条终局路线？
- 是否列出所有新增/复用状态键和对象路径？
- 是否验证“代码存在”和“运行时已登记”不是同一件事？
- 文本语调是否符合所属内容池？
- 是否兼容未拜师、已拜师、叛师记录和剧情中途更换师承，而不修改任何既有师门状态？
- 主要反派是否有明确恶意、主动计划、具体受害者和可见失败？
- 身份反转是否提前给出至少两条独立线索，并能解释此前行为？
- 每章是否至少有一个不依赖战斗、外部知识、随机数或隐藏属性的公平谜题？
- 谜题是否支持日志回看、分级提示、错误反馈和中断恢复？

## 8. 首个落地样板建议

首个样板采用“一纸乡书”，而不是把拜师过程做成强制主线。它可以低风险串联现有资产：

- 新角色从阎罗殿投胎后收到同乡求助，老角色可在扬州补做；
- 出生地或世家决定信件来源和少量线索差异；
- 无师门、已有师门和中途改换师门共享主流程；
- 在扬州武馆接触基础江湖规则；
- 在公开、官断和私了之间作出第一次自主江湖选择；
- 结尾由传闻或一名 AI 玩家第一次叫出玩家的江湖身份。

详细设计见 `04-首章样板-一纸乡书.md`。这条样板能一次验证统一剧情状态、日志、恢复、分支、
AI 回响、出生差异和老存档兼容，并为后续门派章节提供稳定入口。
