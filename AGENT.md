# ZJMud Android Standalone 项目接手说明

> 快照日期：2026-07-17。本文总结项目历史、当前边界以及剧情和 AI 玩家后续路线，供后续开发者与自动化代理接手时使用。事实状态以当前工作区为准；提交基线与未提交实现必须分开描述。

## 1. 项目目标

本项目将 zjmud 服务端、原 Web 客户端和兼容旧 MudOS 语义的 FluffOS 驱动打包为 Android ARM64 单机应用。屏幕前始终只有一名本机玩家；MUD 在不可导出的 `:mud` 前台服务进程中运行，WebView 通过 `127.0.0.1:3000` 与其通信，不依赖外部网络。

这不是原联网服的通用替代品。当前产品保留 MUD 的多对象世界，但主动移除了跨服、充值、运营后台和外部 socket 能力，并针对个人单机体验调整了存档、成长倍率和管理员权限。

## 3. 当前工程状态

### 3.1 已落地并进入提交历史

- Android ARM64 Debug APK 可构建、安装、注册、登录、建角并进入世界。
- FluffOS 运行于独立前台服务进程；JNI 负责启动、日志重定向和受控停止。
- Web 客户端改为本地传输，保留原 ESC/ZJ 控制码与 GB18030 游戏协议。
- runtime 从固定哈希的 `~/zjmud-main.zip` 确定性生成；mudlib 改动以 `tools/` 中的补丁、替换文件和导入规则维护。
- 角色永久属性、递归物品树、动态属性、装备和安全位置可跨退出、进程重建和覆盖升级恢复。
- 5 名 AI 玩家能够持久生活、区域内巡游、休息、购物、交谈、受控会面、有限自卫和安全恢复。
- 已有启动、行为、战斗、迁移、进程终止、生命周期和稳定性测试工具。

### 3.2 当前工作区成果

本次状态快照包含两组重要成果：

1. **AI v2.4 区域旅行**：`tools/mudlib/ai-travel.c`、`docs/AI_TRAVEL_SCHEMA.md`、管理员命令、导入断言及旅行/恢复测试。区域图固定为 4 个区域、8 个节点、14 条有向边和 3 条注册路线；真实跨区日程只批准 `ai_qingfeng`。
2. **前十章 V2 剧情设计**：`story-design/` 中“乡书篇”“名帖篇”和“无面局”总纲。设计稿已经定义反派、反转、谜题、分支预算、状态键与恢复要求，但 runtime 中没有 `plotd`、章节守护、剧情场景或对应测试。

因此描述版本时必须使用以下口径：

- AI 具身与反应基线是 **v1.8 已冻结**。
- AI 跨区域旅行是 **v2.4 已冻结并通过验收**。
- 剧情是 **设计完成到前十章、实现尚未开始**，不能宣称已上线。

### 3.3 仍未完成的产品能力

- Storage Access Framework 存档导入/导出。
- Release keystore、正式签名、版本策略、native symbols 和发布流水线。
- 通用 zjmud 导入器；当前只接受固定源快照。
- 剧情运行框架及前十章 LPC 实现。
- AI v3.x 之后的社会记忆、有限同行、任务图和生成式表达。

## 5. 剧情开发计划

### P0：固定技术契约

- 先定义最小 `/adm/daemons/plotd.c`：日志查询、版本化状态、合法阶段推进、幂等结算和恢复。
- 玩家永久状态统一放入 `plot/<plot_id>/...`；`temp/plot/...` 只保存会话态。
- 固定 `version/status/stage/choices/relations/flags/started_at/updated_at/completed_at` 的含义和迁移方式。
- 唯一奖励必须先持久化选择和新阶段，再结算；物品不是进度真相，丢失或背包满时可补领。
- 建立面向 LPC/runtime 的静态断言和设备内管理员自检，不只验证文件存在。

### P1：实现第一章“一纸乡书”纵向样板

- 实现新角色与老角色双入口、三方调查、个人废仓实例、战斗/说服/赎回三种取簿方式、三种最终选择和回响。
- 加入“缺失的一列”公平谜题：线索可回看、有分级提示、答错有解释、重启后可恢复。
- 不接管拜师；未拜师、已拜师、叛师历史、中途换师门和出生字段缺失均可完成。
- 覆盖死亡、逃跑、断线、服务重启、重复确认、背包满、纪念物丢失和 AI 全部不可用。
- 第一章通过后再冻结 `plotd` 最小接口，避免十章同时依赖未验证框架。

### P2：完成“乡书篇”第一至五章

- 按 `origin_letter_01`、`returning_mark_02`、第三至第五章既定 `plot_id` 顺序实现。
- 每章保持约 70%-80% 共同骨干和 20%-30% 局部差异；每章最多一个跨章核心选择。
- 第五章只保留“全簿见证/隐名存证”篇章级选择，其他方法状态用于对白、线索和非战斗解法，禁止组合爆炸。
- 常驻主城、师父、商店和交通不可被剧情永久破坏；重大变化使用个人实例、传闻或可恢复局部状态。

### P3：完成“名帖篇”第六至十章

- 以“无面局”总纲为准接续前篇，逐章落实沈观澜、陆小栓、闻守拙、叶听秋等反派与身份反转线索。
- 每章至少一个不依赖外部知识、随机数、隐藏属性或强制战斗的谜题。
- 第十章完成篇章结算后回到共同世界；分支影响关系、手段、证据和回响，不复制完整地图与主线。

### P4：让 AI 成为可选剧情角色

- 第一阶段只做风闻、提示、现场见证和事后评价；普通 NPC/日志必须提供完全等价的保底信息。
- AI 不持有唯一线索、唯一物品或剧情进度，不进入核心实例战斗，不替玩家作决定。
- 社会记忆成熟后可增加长期称呼和关系回响；任务图成熟后可执行开发者注册的可选支线动作。
- AI 正式角色身份仍需在世界观文档中明确；在此之前只按可替换配角实现，不让主线正史依赖其存在。

## 6. AI_player 开发计划

AI 玩家使用真实玩家对象，但目标是“受约束的世界居民”，不是复刻真人玩家。每个大版本只增加一个自治维度，完成 `x.4` 冻结前不得进入下一大版本。

### 已完成：收口 v2.4

- 运行旅行静态契约、确定性 runtime 导入、Debug 构建、管理员 smoke 和进程终止恢复矩阵。
- 确认资金只扣一次、失败退款、路线删除/禁用、旧/未来 schema 和非法检查点都回到安全状态。
- 保持只有沈清风启用真实跨区日程；不要在冻结收口时顺手开放其余四名角色。
- v2.4 源码、文档、测试和生成 runtime 作为同一兼容性变更维护。

### v3.x：结构化社会记忆

- `v3.0` 定义事实类型、来源、置信度、时间戳、过期、容量、schema 和诊断。
- `v3.1-v3.3` 依次完成单角色闭环、真实互动接入、迁移/冲突/损坏恢复。
- `v3.4` 冻结容量、保留期、关系影响上限和确定性回归。
- 不保存聊天全文，不推断现实身份，不因负面记忆攻击、尾随或阻断主线。

### v4.x：玩家邀请的有限同行

- 只允许一名玩家与一名 AI 的短时、有限范围、可拒绝/取消/恢复契约。
- 覆盖离线、传送、死亡、禁区、进程终止和双方状态不一致。
- AI 不主动邀请、不接受任意命令、不替玩家战斗或完成任务，也不能绕过费用、权限和剧情门槛。

### v5.x：开发者注册任务图

- 任务图由白名单节点、显式前后条件、预算和补偿动作组成，不接受自然语言任意规划。
- 从单角色线性闭环扩展到批准的分支图，并覆盖重复节点、部分提交、世界变化、迁移和恢复。
- 该能力可服务剧情中的可选 AI 行动，但玩家主线必须在 AI 缺失时完整可玩。

### v6.x：可选生成式表达

- 模型只在确定性行为已经决定后负责“怎么说”，永远不决定“做什么”。
- 输入需裁剪和隐私过滤；输出需 schema 校验、超时、预算、缓存、审核和模板回退。
- 模型输出不得进入 `force_me()`，不得决定命令、路径、任务节点、移动、战斗、奖励或存档字段。
- 离线、超时或拒绝输出时，现有状态机和模板对白必须完全正常。

## 7. 永久设计约束

- 一名真人必须能完成所有核心内容；AI 缺失、暂停、损坏或关闭不能锁死剧情。
- AI 不主动 PK、不追杀、不因记忆报复，不获得管理员权限或任意 LPC 执行能力。
- 路径、事件、等待、重试、记忆、任务图和预算全部设置硬上限。
- 跨步骤行为必须有机器可读状态、取消原因、安全检查点、迁移和幂等恢复。
- 通用剧情不修改师承、叛师记录、贡献、师门任务或拜师次数。
- 奖励优先使用身份、关系、权限、纪念物和有限能力，避免在已有高倍率单机环境中继续数值膨胀。
- 分支必须预先声明收束点及合流后保留的差异；没有收束方案的分支不进入实现。
- 不把 Android 版本、AI 能力版本、AI 存档 schema 和剧情版本混为一谈。

## 8. 开发与验证约定

1. 开工前运行 `git status --short`，保留并理解当前工作区中的用户修改，不覆盖无关变更。
2. 剧情先改 `story-design/` 设计稿并完成评审，再写 LPC；改变 AI 大版本目标或安全边界时先改 `docs/AI_DEVELOPMENT_ROADMAP.md`。
3. mudlib 源码改在 `tools/mudlib/` 或可审计 patch 中，再运行 `./tools/import_zjmud.sh ~/zjmud-main.zip` 重建 runtime；禁止直接编辑或解压回写 `zjmud-runtime.zip`。
4. 只改 Android/Kotlin/native 且不涉及 mudlib 时，可直接运行 `./gradlew :app:assembleDebug`。
5. AI v2.4 的快速静态检查运行 `python3 tools/test_ai_travel.py`；Python 工具至少运行 `python3 -m py_compile tools/*.py`。
6. 涉及 runtime/AI 的设备验收按改动范围运行：

```bash
tools/test_ai_player_startup.sh --yes
adb forward tcp:3000 tcp:3000
tools/ai_admin_smoke.py --travel --scenarios --behaviors
tools/test_ai_player_recovery.py --case travel
tools/test_ai_player_stability.py --duration 1800 --interval 60
```

`test_ai_player_startup.sh --yes` 会清除 `com.zjmud.android` 应用数据，是破坏性测试。剧情框架落地后应提供同等级的确定性状态/迁移测试、管理员自检和进程终止恢复用例，再宣称章节完成。
