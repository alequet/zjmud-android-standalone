# 实施计划

## 1. 工程评估

这是一个中等偏高风险的移植项目，主要工作不是 Android WebView，而是旧版 FluffOS
在 Android NDK/Bionic 上的构建、生命周期改造和遗留编码处理。

单人全职、熟悉 Android 和 C/C++ 的工程师，建议按以下量级安排：

| 阶段 | 预计工程量 | 结果 |
| --- | ---: | --- |
| 基线冻结与资源审计 | 2-3 人日 | 可重复的源包清单和桌面行为基线 |
| FluffOS ARM64/NDK 原型 | 5-10 人日 | Android ARM64 驱动可启动 mudlib |
| Android 运行容器 | 4-7 人日 | 前台服务、JNI、资源部署和状态控制 |
| Web 本地传输改造 | 3-5 人日 | 原 UI 可完成注册、登录和游戏 |
| 持久化、导入导出、恢复 | 3-5 人日 | 数据生命周期完整 |
| 设备测试与收敛 | 5-9 人日 | 可侧载 Release APK |
| 合计 | 22-39 人日 | 约 4-8 周单人开发 |

第一份能登录游戏的技术原型预计需要 8-15 人日。估算不包含大规模修复原 mudlib 自身
已有问题，也不包含重新设计 Web UI。

## 2. 阶段计划

截至 2026-07-13 的实现进度：

| 阶段 | 状态 | 已验证结果 |
| --- | --- | --- |
| 阶段 0 | 基本完成 | 唯一源包哈希固定；runtime 连续两次导入哈希一致 |
| 阶段 1 | 完成 | ARM64 FluffOS preload、回环监听和固定握手通过 |
| 阶段 2 | 原型完成 | 独立前台服务、JNI、原子首次部署和日志落盘通过 |
| 阶段 3 | 核心链路完成 | 原 Web UI 登录、建角和进入世界通过 |
| 阶段 4 | 部分完成 | 回环绑定、外部 daemon/package 禁用；导入导出未实现 |
| 阶段 5 | 未开始 | 仍需真机矩阵、长稳、签名和升级验证 |

### 阶段 0：冻结基线

任务：

- 计算 `~/zjmud-main.zip` 和关键文件哈希。
- 记录当前 Windows 版本的启动日志、登录、注册、进入世界和退出行为。
- 记录服务端实际加载目录和所有运行时写入目录。
- 记录一组 GB2312 协议输入输出样本，包含中文、ANSI、地图和按钮控制码。
- 审计 ZIP 非 ASCII 文件名、重复文件、历史备份和无关二进制。
- 生成机器可读的 `source-manifest.json` 和 allowlist/exclude 清单。

完成门槛：从原 ZIP 能确定性生成同一份 payload，且不会误删运行必需文件。

### 阶段 1：FluffOS Android ARM64 可行性原型

任务：

- 在现有 Mac 开发机安装并固定 Android NDK 和 SDK CMake 版本。
- 使用现有 `Pixel_7` ARM64/API 33 AVD 作为第一调试目标。
- 固定上游提交 `b1453d8b4cb909a381bfb00752a89ffe41d5559f`。
- 建立 NDK CMake 构建，不直接依赖宿主机 Autotools 产物。
- 确认 `local_options` 与原 `tangmen.exe` 所需 efun 一致。
- 将 libevent、PCRE、zlib、crypt 等依赖静态链接或构建为随 APK 分发的 ARM64 库。
- 为 `main()` 增加 JNI 入口，并把进程退出与日志重定向行为显式化。
- 使用 Android 专用 `config.ini` 和绝对 mudlib 路径启动。
- 在 ARM64 模拟器或真机上确认监听 `127.0.0.1:3000`。
- 完成启动、连接、握手、优雅停止和再次启动测试。

完成门槛：最小 Android harness 能启动原 mudlib，TCP 客户端收到“版本验证成功”。

### 阶段 2：Android 应用骨架

任务：

- 建立 Kotlin、Gradle、AndroidX WebKit 工程。
- 设置 `arm64-v8a` ABI 过滤和固定 NDK 版本。
- 实现 payload 首次部署、哈希校验、进度和失败恢复。
- 实现独立 `:mud` 前台服务、通知渠道和 JNI 调用线程。
- 使用 AIDL 暴露启动、停止、状态和诊断信息。
- 实现端口就绪检测，而不是固定等待。
- 捕获 stdout/stderr、FluffOS `debug.log` 和 native tombstone 可用信息。

完成门槛：应用可重复启动/停止服务，主进程和服务进程任一退出时另一方行为明确。

### 阶段 3：Web UI 离线接入

任务：

- 从源包导入 `web/www`，排除 Socket.IO 服务端和 `node_modules`。
- 使用 `WebViewAssetLoader` 加载本地页面。
- 实现 `local-transport.js` 的 Socket.IO 风格兼容接口。
- 实现 Kotlin TCP 桥接、状态机、串行写入和有状态 GB18030 解码。
- 删除外部 CDN、充值、主页和远程 API 代码路径。
- 配置 WebView 导航、文件访问、混合内容和调试开关。
- 处理状态栏、安全区域、软键盘、旋转和窗口恢复。

完成门槛：飞行模式下完成注册、登录、创建角色、进入世界和常用游戏操作。

### 阶段 4：离线约束与数据生命周期

任务：

- 驱动强制绑定回环地址。
- 禁用或限制 LPC socket efun 和网络 daemon。
- 实现玩家数据导出、导入、备份、校验和回滚。
- 定义 APK 更新时的 payload 与可写数据合并策略。
- 实现显式“重启服务”“停止服务”“重置游戏”操作。
- 增加崩溃重启限流和损坏状态检测。

完成门槛：更新、异常退出、导入失败和存档恢复均不会静默破坏数据。

### 阶段 5：兼容性与发布

任务：

- 在至少三类 ARM64 设备和多个 Android 大版本上测试。
- 验证后台、熄屏、低内存、服务进程崩溃、WebView 进程回收和系统升级场景。
- 做全离线抓包和外部 URL 拦截测试。
- 记录启动时间、内存、CPU、耗电和 APK/安装体积。
- 关闭 Release WebView 调试，生成签名 APK 和版本清单。
- 完成安装、升级、降级限制和存档迁移说明。

完成门槛：通过 `TEST_PLAN.md` 的发布门槛并产出可重复签名构建。

## 3. 关键路径

```text
源包审计
  -> ARM64 FluffOS 启动
  -> JNI 独立服务进程
  -> 本地 TCP/GB18030 桥接
  -> 原 Web UI 登录成功
  -> 生命周期与数据恢复
  -> 多设备验证
```

在 FluffOS 真机启动和 GB18030 往返通过之前，不应投入大量时间重构 Web UI。

## 4. 建议的里程碑

- M1：ARM64 命令行驱动在 Android 环境启动。
- M2：最小 APK 启动服务，Telnet 测试客户端完成握手。
- M3：WebView 进入游戏，完成第一轮端到端操作。
- M4：飞行模式、后台运行和进程恢复通过。
- M5：存档导入导出、更新保护和 Release 构建通过。

## 5. 完成定义

项目完成必须同时满足：

- APK 内含全部必要运行资源，安装后无需下载。
- 设备外网络完全不可达且不会产生尝试流量。
- 原 Web UI 的主要游戏功能无协议回归。
- 服务进程异常不导致主 UI 直接崩溃。
- 玩家数据在重启、升级和受控恢复中保持完整。
- 构建输入、版本、哈希和签名步骤都有文档并可重复执行。
