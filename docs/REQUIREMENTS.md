# 产品与验收要求

## 1. 产品目标

将已经调通的 `~/zjmud-main.zip` 整体转化为一款纯单机 Android 应用：

1. 启动应用即启动本地 zjmud 服务端。
2. 应用主界面呈现压缩包中的原 Web UI。
3. 前端自动连接应用内部的服务端，无需填写地址和端口。
4. 游戏运行、登录、注册、存档和恢复均不依赖设备外网络。
5. 应用进入后台或熄屏后，服务端通过前台服务继续运行，直到用户明确停止。

## 2. 范围

### 2.1 首期必须包含

- Android ARM64 APK。
- FluffOS ARM64 运行时。
- zjmud mudlib、初始数据和必要的运行目录。
- 原 Web UI 的 HTML、CSS、JavaScript、图片和字体资源。
- WebView 到本地 Telnet 服务的双向 GB18030/GB2312 传输层。
- 服务启动、就绪检测、异常恢复、停止和日志采集。
- 玩家数据持久化。
- 存档导入、导出和重置入口。
- 全离线运行验证。

### 2.2 明确不包含

- 多人联机和局域网访问。
- 公网账号、充值、网站主页、统计、更新检查和远程日志。
- Play 商店发布、多 ABI、云存档和账号同步。
- 对原游戏内容、数值或 UI 的重新设计。
- 继续使用 Windows/Cygwin 的 `tangmen.exe`。

## 3. 功能要求

### 3.1 启动

- 首次启动应显示明确的资源准备状态。
- 部署过程必须可中断且可恢复，不能留下被误认为完整安装的半成品目录。
- 服务端监听成功后再开放 Web UI 的登录交互。
- 常规启动不得重复覆盖玩家数据。

### 3.2 游戏连接

- 服务端只监听 `127.0.0.1:3000` 或运行时分配的本地端口。
- 客户端不提供服务器地址编辑入口。
- 连接建立后按原代理行为发送 `zjmDMaIpOvxdb\n\n`。
- 服务端输出采用有状态解码器处理，必须正确处理跨 TCP 包拆分的中文多字节字符。
- Web UI 发出的 Unicode 命令编码为 GB18030 后发送给服务端。
- 断线后 UI 应显示本地服务状态，并支持自动重连或一键重启服务。

### 3.3 Web UI

- 保留原页面的布局、命令按钮、地图、状态栏、聊天区和 ANSI 颜色解析。
- 用本地传输对象替换 `io.connect()`，保持 `sock.on()` 和 `sock.emit()` 的调用接口。
- 移除 Socket.IO 远程脚本依赖。
- 移除 `cdn.bootcss.com`、`92mud.com` 等外部请求和跳转。
- 充值、游戏主页等在线入口应删除或替换为适合单机版的本地操作。
- WebView 只允许 `appassets.androidplatform.net` 和应用自身页面导航。

### 3.4 生命周期

- 服务端运行在独立 Android 进程，原生崩溃不能直接终止 WebView 主进程。
- 服务运行期间显示不可误解的前台服务通知。
- 应用切到后台或熄屏时服务继续运行。
- 用户从通知或应用菜单选择“停止服务”时执行优雅关闭。
- 用户强制停止应用后不尝试绕过 Android 行为自动复活。
- 设备重启后默认不自启动；用户再次打开应用时恢复服务。

### 3.5 数据

- mudlib 和玩家数据位于应用私有存储，不申请传统外部存储权限。
- APK 更新不得覆盖 `data`、`backup`、`dump`、`log` 和其他已确认的可写目录。
- 存档导出生成单一归档文件，并带格式版本与校验值。
- 导入前自动创建当前存档备份；导入失败必须回滚。
- 重置操作需要二次确认并先提供导出机会。

## 4. 非功能要求

- 安装后可在飞行模式下完成首次启动、注册、登录、游戏和重启恢复。
- 首期只支持 `arm64-v8a`，最低 Android 版本在原型阶段验证后固定，建议从 API 26 起评估。
- 构建过程必须固定 FluffOS 提交、NDK 版本、依赖版本和资源包 SHA-256。
- Release APK 不包含 Windows EXE/DLL、Node.js `node_modules`、`.DS_Store` 和无关历史压缩包。
- Native 崩溃、FluffOS 标准输出和 `debug.log` 应可从应用内导出。
- 不记录或发送遥测数据。

## 5. 验收场景

1. 全新安装，开启飞行模式，启动应用并成功进入登录页。
2. 注册本地角色、进入世界、执行移动/战斗/存档相关命令。
3. 杀死主 UI 进程后重新打开，服务状态和玩家数据正确。
4. 服务进程被系统终止后，前台服务按策略重建并可重新连接。
5. 熄屏一段时间后恢复，游戏服务仍可连接；若被厂商策略终止，应给出可恢复状态。
6. 更新 APK 后玩家数据不丢失。
7. 导出存档、清空应用数据、重新安装并导入，角色状态恢复。
8. 抓包确认没有发往非回环地址的 TCP、UDP、DNS 或 HTTP 流量。
9. WebView 尝试打开外部 URL 时被应用拦截。
10. 服务端异常退出时 WebView 不崩溃，并显示重启入口。
