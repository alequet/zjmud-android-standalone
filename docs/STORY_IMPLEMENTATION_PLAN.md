# 前五章剧情落地实施计划

## 1. 目标与范围

本计划把“乡书篇”前五章落地到当前 Android 单机版 zjmud。剧情设计以
`story-design/16-前十章重构总纲-无面局.md` 和各章末尾的 V2 小节为准；各章早期版本中与
V2 冲突的反派、谜题和身份描述不进入实现。

本轮工程目标不是一次提交五章全部内容，而是按可独立游玩、可验证、可回滚的版本逐章交付：

1. 建立章节剧情平台和持久化闭环；
2. 完成第一章垂直切片；
3. 用第二章验证跨章分支收束；
4. 实现第三、第四章的共用调查与实例能力；
5. 实现第五章高潮和篇章结算；
6. 完成全篇升级、恢复和兼容回归。

不在前五章落地范围内：重做建角、改变拜师规则、改造师门任务、让 AI 承担关键进度、开发原生
Android 剧情页面、实现第六章以后内容。

## 2. 当前基础与架构结论

### 2.1 可复用能力

- 玩家 dbase 和 `save()` 可直接持久化 `plot/*` mapping，旧角色无需批量改档；
- Android runtime 升级会保留 `data/`、`backup/`、`log/` 等可变目录；
- ZJ 协议已有长文本、动作菜单、输入、确认框和可点击命令；
- 扬州已有杜宽、谭友纪、武庙、客栈、北门和衙门等适合增加薄入口的常驻资产；
- `fubend` 的批量克隆与出口重写可以参考，但不能直接作为剧情实例守护。

### 2.2 必须新增的能力

现有 `questd` 面向重复师门任务，`storyd` 面向定时全局奇遇，都不能承担章节主线。新增：

- `/adm/daemons/plotd.c`：剧情注册、状态初始化、阶段推进、选择、关系、日志和幂等结算；
- `/adm/daemons/plot/*.c`：每章规则与展示；
- `/cmds/usr/plot.c`：玩家剧情日志入口；
- `/cmds/adm/plotadmin.c`：检查、修复、阶段测试和自检；
- 剧情实例管理：玩家隔离、所有权校验、安全退出、服务重启后按持久阶段重建；
- 公共组件：关键物品补领、谜题提示、二次确认、动态战斗分档、AI 缺席降级。

### 2.3 源码与发布边界

剧情 LPC 以 UTF-8 保存在 `tools/mudlib/plot/`，`tools/import_zjmud.sh` 导入时转换为 GB18030。
不得直接编辑 `app/src/main/assets/runtime/zjmud-runtime.zip`。对上游既有文件的薄修改使用补丁保存，
导入后进行文件、预载、命令入口和网络 efun 校验。

### 2.4 原创剧情战斗约束

原创剧情战斗统一遵守 `docs/ORIGINAL_PLOT_COMBAT_SPEC.md`。开发时冻结以下规则：

1. 当前实力主要按 `combat_exp` 和 zjmud 原生有效战斗等级判断，奇经八脉只用于同经验档内适度加强；
2. 不以目标战斗时长作为敌人设计锚点；
3. Boss 战允许正常产生玩家或敌人死亡，副本死亡沿用当前 zjmud 机制；
4. 敌人可以使用玩家未接触的武学，武学、属性、兵器或徒手配置参考原生副本；
5. 高等级玩家只在重点 Boss 战增加一到两个阶段，并让 Boss 实力阶梯式上涨一到两个层级；
6. 奖励剧情化，不掉落随机强装备，不绕过玩家自制装备生态；
7. 原创 Boss 战默认单人完成，暂不让 AI 玩家参与核心战斗。

重点角色另受固定武学背景约束：门派或势力来源、完整映射、徒手准备、兵器和阶段绝招均来自剧情
白名单，实时生成只缩放等级与战斗属性。每阶段至少满足核心 `perform` 的技能、基本内功和内力前置；
不得随机换门派，也不得因实例恢复漂移。武学来历不强行写入主线，只在不破坏原剧情时旁敲侧击。
闻守拙的正式背景冻结为丐帮旧支。

每个高潮副本必须为剧情地点单独设计房间图。可以参考 `fubend` 的克隆、出口重写、清理和撤出机制，
但不得复用现有 `/d/fuben/*` 房间布局，也不得把剧情唯一进度交给原生副本对象或掉落物。

所有原创剧情 NPC 统一遵守 `docs/ORIGINAL_PLOT_NPC_LIFECYCLE.md`：副本对象随实例销毁，
世界联系人由常驻房间刷新，关键证人同时处理 `unconcious()` 和 `die()`，有后续连续性的 Boss
使用非致死落败并在销毁克隆前持久化命运。该文档是整个原创剧情系统的全局开发约束，不局限于前三章。

## 3. 冻结的技术契约

### 3.1 稳定 ID

- 篇章 ID：`hometown_letters_01`；
- 第一章：`origin_letter_01`；
- 第二章：`returning_mark_02`；
- 第三章：`silent_crossing_03`；
- 第四章：`visitors_from_home_04`；
- 第五章：`where_rivers_end_05`。

设计稿中的 `hometown_letters_arc_01` 视为旧别名，不写入新存档。第三章登船路线的固定收束阶段
统一为 `reach_cargo_hold`，不使用总纲中出现的 `open_hidden_hold`。

### 3.2 玩家状态

```text
plot/<plot_id>/version
plot/<plot_id>/status          locked/available/active/completed
plot/<plot_id>/stage
plot/<plot_id>/started_at
plot/<plot_id>/updated_at
plot/<plot_id>/completed_at
plot/<plot_id>/choices/*
plot/<plot_id>/relations/*
plot/<plot_id>/flags/*

plot/arc/hometown_letters_01/version
plot/arc/hometown_letters_01/status
plot/arc/hometown_letters_01/current_chapter
plot/arc/hometown_letters_01/choices/*
plot/arc/hometown_letters_01/relations/*
plot/arc/hometown_letters_01/flags/*
plot/arc/hometown_letters_01/completed_at
```

规则：

- `temp/plot/*` 只保存当前页面、战斗和对象引用；
- 状态写入后立即调用玩家 `save()`；
- 唯一选择和结算先落状态再播放文本、发物品；
- 实体证物不是进度真相，可按状态补领；
- 版本迁移惰性执行，不遍历历史存档；
- 师门、转世、正负神、AI 在线状态不得成为章节主进度条件。

### 3.3 调用安全

客户端只能提交稳定动作 ID，例如：

```text
plot act origin_letter_01 inspect_ledger
```

动作由章节守护白名单分派。客户端不得传 LPC 文件路径、对象名、奖励 ID、下一阶段或任意命令。
`plotd` 的写接口只接受已登记章节守护调用。

### 3.4 实例恢复

- 实例不保存对象引用和 `call_out`；
- 玩家状态只记录章节、阶段、已完成事实和安全返回点；
- 剧情房间设置 `no_save_location`，登录位置始终回退到安全常驻房间；
- 重新进入时由阶段重建房间、NPC 和机关；
- 战斗失败只重置战斗事实，不回退已持久化调查和谜题答案；
- 实例损坏时必须存在不依赖关系、师门或物品的保底入口。

## 4. 分阶段实施

### 阶段 0：技术契约与工程骨架（2 至 3 人日）

交付：

- 本实施计划；
- 稳定 ID、阶段名、状态 schema 和调用边界；
- UTF-8 剧情源码目录；
- 导入、编码转换、预载和静态验证链。

验收：连续运行两次导入结果一致；runtime 中剧情文件为 GB18030；`plotd` 只预载一次；不存在
客户端可控路径调用。

### 阶段 1：剧情平台最小闭环（7 至 10 人日）

交付：

- `plotd` 注册、查询、初始化、阶段推进、选择和完成接口；
- `plot` 日志菜单及 `plotadmin` 自检；
- 一个内部测试章节走通“可用 → 接取 → 推进 → 选择 → 完成”；
- 剧情实例创建、退出、清理、重建的最小实现；
- telnet 自动化测试和强停恢复测试。

验收：在任一关键阶段强停 mud 进程并重启后可以恢复；重复动作不重复推进或领奖；实例不能被
其他角色进入；完成状态可随 APK runtime 升级保留。

### 阶段 2：第一章“一纸乡书”（9 至 13 人日）

交付：

- 新角色短笺、老角色武庙/杜宽双入口；
- 八类出身适配和籍贯无考兼容；
- 许三娘、周守义及三方调查入口；
- 废仓实例、裴九战斗/说服/赎回三种取得方式；
- “缺失的一列”谜题、分级提示和错误反馈；
- 三种账簿去向、关系写入、纪念物和回响。

验收：所有身份组合可完成；中途拜师/换师门不改变进度；死亡、逃跑、断线、背包满和 AI 全部
缺席均不锁死；选择及奖励严格幂等。

### 阶段 3：第二章“旧印新痕”（6 至 9 人日，已完成）

交付：

- 按首章选择生成三种同构短入口；
- 第一章关系幂等映射到篇章状态；
- 旧仓、孟四、青篷客和“双半印”谜题；
- 第一版公共剧情敌人生成器、第二章专属旧仓房间图和青篷客单人战斗实例；
- 第三章青芦渡入口。

验收：三种首章结局均在本章前 25% 的 `compare_marks` 合流；首章关系字段缺失时走杜宽兼容
入口；本章不产生新的跨章核心选择；青篷客覆盖经验档、奇经八脉增强、徒手自适应生成、死亡恢复
和无 AI 单人完成。当前已通过三分支 × 三种处置方式运行矩阵、六档强度阶梯、真实战斗与玩家原生
死亡、逃离重入、孟四保护、环境前置、奖励幂等、双半印错配反馈和章节结算；敌人存活、敌人死亡
未转场、案台部分解谜和完成结算四个 Android 强停断点均已通过。

### 阶段 4：第三、四章（13 至 19 人日，第三章已完成）

第三章交付夜船灯号、三种登船方式、夜船实例、阿禾救援、三类货物、焚证冲突和船户关系。
所有登船方式在 `reach_cargo_hold` 合流。当前已接入甲板、船首、船尾、狭舱通道、货舱和舱底
六房个人实例；正面截船需要清除甲板封锁，汲水灭火需要往返舱底，其他路线不设技能硬门槛。

第四章交付八种出身文本皮肤、三封乡书、假陆小栓、三种短行动、花鹞子追踪、请帖解码和证据
准备。所有行动在 `catch_hua_yaozi` 合流。

验收：核心谜题不依赖外部知识；错误答案给出具体冲突；移动端慢操作和断线不造成关键人物永久
死亡；身份反转的独立线索均可在日志回看。

### 阶段 5：第五章“百川归处”（9 至 14 人日，已完成）

交付：

- 四种入场便利及保底排水道；
- 多阶段旧盐仓实例、真陆小栓救援、护簿冲突和纵火阶段；
- 五份命令的双笔迹谜题与闻守拙/沈观澜身份分离；
- `archive_disposition` 二次确认、两种尾声、篇章纪念页和回顾；
- 第六章共用入口状态，但不实现第六章内容。

验收：无关系、无师门和缺少附加证据均可完成；战胜闻守拙不能跳过谜题和总簿选择；两种选择
回到相同开放世界且奖励等价；尾声中断后只补播和补领。

### 阶段 6：全篇加固（5 至 8 人日）

- 全身份、全选择、全解法和全恢复矩阵回归；
- 新存档、历史存档和上一 APK 版本升级验证；
- GB18030 文本、ZJ 控制包、移动端按钮长度和长日志检查；
- 战斗异常长短观察、对象泄漏、实例清理和存档体积检查；战斗时长不作为强度锚点；
- 文案语调、世界命名、师门自由和转世脱钩审查。

阶段 6 的自动化入口固定为 `tools/plot_arc_hardening.py`：默认执行 54 个剧情源码的
GB18030 往返、runtime 压缩包逐文件一致性、ZJ 菜单命令和按钮长度、世界/身份边界、实例
清理契约检查；加 `--runtime` 后使用专用 QA 角色覆盖无 `plot/` 历史存档、缺失 `born`、
无师门/有师门/叛师记录、schema v1 合法迁移、损坏状态修复和未来 schema 原子拒绝。
`plotadmin audit` 输出剧情状态字符数、实例临时引用和驱动对象数，用于升级、完成结算和
重复进入后的泄漏观察。剧情状态 schema 已提升到 v2：合法中途进度保留，未完成损坏章节
回到安全初始阶段，已完成章节保留完成并归一到终局阶段，未来版本只读拒绝。

## 6. 测试矩阵

每章至少覆盖：

- 新角色、老角色、无师门、已有师门、有叛师历史、中途换师门；
- 八个出身内容组和出生字段缺失；
- 新手、中期、顶尖战斗档位；
- 每种战斗、说服、机关、追踪和正式选择；
- 关键阶段断线、死亡、退出、服务强停和 runtime 升级；
- 背包满、证物丢失、NPC 加载失败、AI 暂停；
- 重复点击、重复谜题提交、重复选择确认和重复领取；
- 第一章三结局进入第二章并收束；第五章两结局进入共同后续状态。

自动化分层：

1. 静态测试：注册表、阶段引用、文件编码、路径白名单和导入产物；
2. LPC 自检：状态迁移、非法阶段、重复推进、非法选择和版本兼容；
3. telnet 流程：真实登录、菜单输出、接取/推进/重启/结算；
4. Android 仪器测试：runtime 升级、服务强停恢复和 WebView 菜单；
5. 人工实机：谜题可读性、按钮长度、战斗阶段、武学/兵器表现、地点一致性和文案回响。

## 7. 发布策略与完成定义

每一章作为独立 runtime 版本发布。上线下一章前，上一章必须满足：

- 章节主线和所有正式选择可完成；
- 自动化覆盖全部持久阶段；
- 无重复奖励、跨角色实例访问或客户端路径注入；
- 强停和升级恢复通过；
- AI 全部不可用时剧情完整；
- 设计状态键、实现注册表和测试夹具一致；
- 当前版本可从剧情日志明确看到目标、证据、提示和完成回顾。

总工作量预计 51 至 76 人日。第一章可玩测试版预计在阶段 0 至 2 完成后交付，即约 18 至 26
人日。工作量不包含第六章以后内容和大规模重写既有扬州地图。
